[Proposed] Undo/Redo feature

Proposed Implementation

The undo/redo mechanism is facilitated by UndoRedoManager. It contains a history (of EventList states), stored internally as an eventListStateList, as well as a currentStateIndex, which contains the index of the current state. UndoRedoManager also contains a mainEventList, which is the EventList that the GUI is in sync with.

It is important that the Model works only with the mainEventList (and not duplicates) for this reason. To achieve this, eventListStateList contains deep-copies of the mainEventList, which are created and saved to eventListStateList each time a state-changing command that is also not the undo or redo command (such as add_task or delete_task) is being executed. These state-changing commands that are neither undo or redo commands will modify mainEventList directly before the deep-copies of the modified mainEventList are created and stored.

Every time an undo or redo occurs, the mainEventList is reverted to a past or future state of itself by retrieving its data from the appropriate copy in the list of duplicates. This allows us to always work with the same EventList instance throughout the program execution, which is crucial since this particular instance is in sync with the GUI; only changes to this instance are reflected as changes to the GUI.

In addition, UndoRedoManager implements the following operations:

  • UndoRedoManager#commit() — Creates a deep-copy of the mainEventList state and saves that copy to its history using EventList’s copy-constructor

  • UndoRedoManager#undo() — Retrieve mainEventList’s previous state data from its duplicate and restore mainEventList

  • UndoRedoManager#redo() — Retrieve mainEventList’s future state data from its duplicate and restore mainEventList

  • UndoRedoManager#clearFutureHistory() -- Delete all duplicates of mainEventList that occur after the index given by the currentStateIndex in eventListStateList

These operations are exposed in the Model interface as Model#commitToHistory(), Model#undoFromHistory() and Model#redoFromHistory(), and Model#clearFutureHistory() respectively.

Given below is an example usage scenario and how the undo/redo mechanism behaves at each step.

Step 1. The user runs the program for the first time. The UndoRedoManager will be initialized with the initial mainEventList. A duplicate of mainEventList will be added to eventListStateList and the currentStateIndex will point to that single EventList state in the list.

Step 2. The user executes add_event “Suntec City Computer Fair” --at “17/11/2019 12:00”. This add_event command will first call Model#clearFutureHistory() in order to discard all future states (in case an undo command was called before) as they are no longer useful. In this case, there are no future states to discard. All other state-changing commands that are not undo or redo will discard future states as well when executed. Then, the event will be added to mainEventList. Finally, Model#commitToHistory() will be called, causing the modified mainEventList to be deep-copied and saved to eventListStateList, and the currentStateIndex is incremented by one to contain the index of the newly inserted event list state.

If a command fails its execution, it will not call Model#commitAddressBook(), so the address book state will not be saved into the addressBookStateList.

Step 3. Suppose the user decides that adding the task was a mistake. He/she then executes the undo command to rectify the error. The undo command will call Model#undoFromHistory(), which will decrement the currentStateIndex by one to contain the index of the previous event list state, which mainEventList resets its data to.

If the currentStatePointer is at index 0, pointing to the initial event list state, then there are no previous event list states to restore. The undo command uses Model#canUndoEventList() to check if this is the case. If so, it will return an error to the user rather than attempting to perform the undo.

The redo command does the opposite — it calls Model#redoFromHistory(), which increments the currentStateIndex by one to contain the index of the previously undone state. The mainEventList then resets its data to this state.

If the currentStateIndex is at index eventListStateList.size() - 1, pointing to the latest event list state, then there are no undone event list states to restore. The redo command uses Model#canRedoEventList() to check if this is the case. If so, it will return an error to the user rather than attempting to perform the redo.

Step 4. The user decides to execute the command list. Non-state-changing commands such as list do not manipulate task and event data. Hence, these commands will not call Model#clearFutureHistory() or Model#commitToHistory() no changes to the current state have been made. Thus, the eventListStateList remains unchanged.

Step 5. The user executes delete_event 1, which calls Model#clearFutureHistory(), deletes the event from mainEventList, and then calls Model#commitToHistory(). As mentioned in Step 2, all event list states after the currentStateIndex will be purged.

Design Considerations

Aspect: How undo & redo executes

  • Alternative 1 (current choice): Saves the entire address book.

    • Pros: Easy to implement.

    • Cons: May have performance issues in terms of memory usage.

  • Alternative 2: Individual command knows how to undo/redo by itself.

    • Pros: Will use less memory (e.g. for delete, just save the person being deleted).

    • Cons: We must ensure that the implementation of each individual command are correct.

Aspect: Data structure to support the undo/redo commands

  • Alternative 1 (current choice): Use a list to store the history of address book states.

    • Pros: Easy for new Computer Science student undergraduates to understand, who are likely to be the new incoming developers of our project.

    • Cons: Logic is duplicated twice. For example, when a new command is executed, we must remember to update both HistoryManager and VersionedAddressBook.

  • Alternative 2: Use HistoryManager for undo/redo

    • Pros: We do not need to maintain a separate list, and just reuse what is already in the codebase.

    • Cons: Requires dealing with commands that have already been undone: We must remember to skip these commands. Violates Single Responsibility Principle and Separation of Concerns as HistoryManager now needs to do two different things.
